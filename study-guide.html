Domain 1: Describe Cloud Concepts

25–30% of the exam — Cloud computing fundamentals, benefits, and service types

What is Cloud Computing?

Cloud computing is the delivery of computing services over the internet. Instead of buying and maintaining physical servers, storage, and networking equipment, you rent them from a cloud provider like Microsoft Azure.

Computing services include:

  • Compute — Virtual machines, containers, serverless functions
  • Storage — Files, blobs, disks, databases
  • Networking — Virtual networks, load balancers, DNS
  • Databases — SQL, NoSQL, caching
  • AI & Analytics — Machine learning, cognitive services, big data
  • And much more — IoT, DevOps, security tools

Five Characteristics of Cloud Computing (NIST)

  1. On-demand self-service — Provision resources without human interaction with the provider
  2. Broad network access — Available over the network from any device
  3. Resource pooling — Provider's resources are shared across multiple customers (multi-tenant)
  4. Rapid elasticity — Resources can be scaled quickly based on demand
  5. Measured service — Usage is monitored, metered, and billed transparently
The exam may test whether you can identify which characteristics belong to cloud computing vs traditional on-premises IT.

The Shared Responsibility Model

Security and management in the cloud is a shared effort between the cloud provider and the customer. Who is responsible for what depends on the type of service you use.

Key Idea: The cloud provider is always responsible for the physical infrastructure. The customer is always responsible for their data, accounts, and devices. Everything in between depends on the service type.

What the Cloud Provider Always Manages:

  • Physical datacenter (building, power, cooling, physical security)
  • Physical network (cabling, switches, routers)
  • Physical hosts (servers, hardware)

What the Customer Always Manages:

  • Data & information stored in the cloud
  • Devices that connect (phones, PCs)
  • Accounts & identities (user access management)

How Responsibility Shifts by Service Type:

Remember: As you move from IaaS → PaaS → SaaS, more responsibility shifts to the provider. But data and identity are ALWAYS the customer's responsibility.

Cloud Deployment Models

There are three main ways to deploy cloud resources:

1. Public Cloud

  • Owned and operated by a third-party provider (Microsoft, AWS, Google)
  • Resources are shared across multiple organizations (multi-tenant)
  • Accessed over the internet
  • No upfront CapEx — pay-as-you-go
  • Highest scalability, lowest management burden

2. Private Cloud

  • Dedicated to a single organization
  • Can be hosted on-premises or by a third party
  • Organization has full control
  • Higher cost, more maintenance
  • Best for strict compliance or security requirements

3. Hybrid Cloud

  • Combines public and private cloud (or on-premises)
  • Data and applications can move between environments
  • Maximum flexibility — keep sensitive data private, burst to public cloud for extra capacity
  • Most common in enterprise environments

Multi-cloud means using multiple public cloud providers (e.g., Azure + AWS) to avoid vendor lock-in or leverage different strengths.
Know the differences between public, private, and hybrid cloud. The exam loves scenario questions asking which model fits a given situation.

Benefits of Cloud Computing

The exam tests your understanding of each benefit. Learn what each one means and be able to match it to a scenario.

High Availability

Services remain available with minimal downtime, even when failures occur. Measured by uptime percentages in SLAs.

  • 99.9% uptime = ~8.7 hours downtime/year
  • 99.99% uptime = ~52 minutes downtime/year

How Azure achieves it: Availability Zones, geo-redundancy, load balancing, auto-failover.

Scalability

The ability to add or remove resources to handle changes in demand.

  • Vertical scaling (scale up/down) — Add more CPU or RAM to an existing VM
  • Horizontal scaling (scale out/in) — Add or remove VM instances

Vertical = making one machine bigger. Horizontal = adding more machines.

Elasticity

Resources are scaled automatically in response to demand. You don't have to predict traffic — the system adjusts.

Example: An e-commerce site automatically scales up during Black Friday and scales back down after.

Scalability = the ability to scale. Elasticity = scaling happens automatically.

Reliability

The ability of a system to recover from failures and continue functioning.

Cloud's decentralized design (resources in multiple regions) supports reliable infrastructure. Even if one region goes down, others keep running.

Predictability

Two aspects:

  • Performance predictability — Autoscaling, load balancing, and high availability ensure consistent user experience
  • Cost predictability — Track spending, forecast costs, and use analytics to optimize budget
Security

Cloud providers invest billions in security. They offer tools like firewalls, identity management, encryption, and threat detection. However, security is still a shared responsibility — you must configure and manage your own access controls and data.

Governance

Cloud features like templates, policies, and auditing help ensure your resources comply with corporate standards and regulations. Policies can flag or even auto-fix resources that fall out of compliance.

Manageability

Two meanings:

  • Management OF the cloud — Auto-scaling, template-based deployment, automated monitoring and alerts
  • Management IN the cloud — Multiple tools: web portal, CLI, PowerShell, APIs
Agility

Deploy and configure resources rapidly. What used to take weeks (ordering hardware, racking servers) now takes minutes in the cloud.

Disaster Recovery

Replicate data and apps to a secondary region so that services can be restored quickly after a major event (natural disaster, data corruption, etc.).

Fault Tolerance

A system continues operating even when individual components fail. Achieved through redundancy — having backup components ready to take over.

Geo-distribution

Deploy apps and data to regions around the world, ensuring users everywhere get fast performance and the system can survive regional outages.

The exam often asks about the financial benefits of cloud. Know that cloud shifts CapEx to OpEx and follows a consumption-based model.

Think of it like pizza:

  • On-premises = You make everything from scratch at home
  • IaaS = You buy dough and ingredients (infrastructure), you cook and serve
  • PaaS = You get a pre-made pizza to customize and bake (platform handles the base)
  • SaaS = Pizza delivered to your door (ready to eat)

35–40% of the exam — The biggest domain. Core components, compute, networking, storage, identity, and security

Core Architectural Components

Azure Organizational Hierarchy

Azure resources are organized in a hierarchy. Policies and access applied at higher levels flow down to lower levels.

Management Groups
→
Subscriptions
→
Resource Groups
→
Resources
Management Groups
  • The top level of organization (above subscriptions)
  • Apply policies and RBAC across multiple subscriptions
  • Can be nested up to 6 levels deep
  • One root management group per Azure AD tenant
  • Example: An enterprise might have management groups for "Production" and "Development," each containing multiple subscriptions
Subscriptions
  • A billing boundary — separate bill per subscription
  • An access control boundary — manage permissions per subscription
  • An account can have multiple subscriptions
  • Common pattern: separate subscriptions for dev, test, and production
Resource Groups
  • Logical containers for related Azure resources
  • Every resource must be in exactly one resource group
  • Resources can interact across resource groups
  • Deleting a resource group deletes ALL resources in it
  • Cannot be nested
  • Tags are NOT inherited by resources inside the group (by default)
Resources

Anything you create in Azure is a resource: a VM, a database, a storage account, a virtual network, etc. Each resource exists in exactly one resource group.

Azure Regions

  • A geographical area containing one or more datacenters
  • Azure has 60+ regions across the globe
  • You choose the region when deploying resources (affects latency, compliance, pricing)
  • Not all services are available in all regions

Region Pairs

  • Two regions in the same geography paired together (e.g., East US ↔ West US)
  • At least 300 miles apart for disaster recovery
  • Azure updates one region in the pair at a time (no simultaneous downtime)
  • If a region fails, its pair is prioritized for recovery

Availability Zones

  • Physically separate datacenters within a single region
  • Each zone has independent power, cooling, and networking
  • Minimum of 3 zones per supported region
  • Protects against datacenter-level failures
  • Provides 99.99% VM SLA

Sovereign Regions

  • Isolated Azure instances for compliance/legal requirements
  • Azure Government — US government agencies (separate physical and logical network)
  • Azure China — Operated by 21Vianet (data stays in China)
Know the hierarchy: Management Groups > Subscriptions > Resource Groups > Resources. Also remember that deleting a resource group deletes everything in it.

Identity, Access, and Security

Microsoft Entra ID (formerly Azure Active Directory)

Microsoft's cloud-based identity and access management service. It is NOT the same as on-premises Active Directory.

On-premises Active DirectoryMicrosoft Entra ID
LocationOn-premises serversCloud
ProtocolsKerberos, LDAP, NTLMSAML, OAuth, OpenID Connect, REST APIs

Key Identity Features

  • Single Sign-On (SSO) — Sign in once, access multiple apps without re-entering credentials
  • Multi-Factor Authentication (MFA) — Requires 2+ verification factors: something you know (password) + something you have (phone) + something you are (biometric)
  • Conditional Access — IF-THEN policies: IF a user meets certain conditions (location, device, risk), THEN require additional verification or block access
  • External Identities — Let people outside your org sign in with their own credentials (B2B for partners, B2C for consumers)
  • Passwordless Authentication — Sign in with Windows Hello, FIDO2 keys, or the Authenticator app (no password needed)

Azure RBAC (Role-Based Access Control)

Controls who can do what on which Azure resources.

  • Applied at a scope: Management Group > Subscription > Resource Group > Resource
  • Permissions inherit downward
  • Built-in roles:
RoleCan Do
OwnerFull access + can assign roles to others
ContributorFull access but CANNOT assign roles
ReaderView only — cannot change anything
User Access AdministratorManage user access to resources
RBAC controls access to Azure resources. Entra ID roles control access to Entra ID objects (users, groups). They are separate systems.

Security Models

Zero Trust

A security model that assumes breach and treats every request as if it comes from an untrusted network.

Three principles:

  1. Verify explicitly — Always authenticate and authorize based on all available data
  2. Use least privilege access — Limit access to the minimum necessary
  3. Assume breach — Minimize blast radius and segment access; verify end-to-end encryption
Defense in Depth

A layered approach to security. If one layer fails, the next one stops the attacker.

Layers (outside in):

  1. Physical security — Datacenter access controls
  2. Identity & access — MFA, Conditional Access, SSO
  3. Perimeter — DDoS protection, firewalls
  4. Network — NSGs, subnets, deny by default
  5. Compute — Secure VMs, patching, endpoint protection
  6. Application — Secure code, vulnerability management
  7. Data — Encryption at rest and in transit
Know the three Zero Trust principles (verify explicitly, least privilege, assume breach) and the Defense in Depth layers.

Governance and Compliance

Azure Policy

Create and enforce rules for your Azure resources at scale.

  • What it does: Evaluates resources and flags or prevents those that don't comply with your rules
  • Policy effects:
    • Audit — Logs non-compliance but doesn't prevent it
    • Deny — Prevents the creation of non-compliant resources
    • Modify — Automatically adds or changes properties (like tags)
    • DeployIfNotExists — Deploys a required resource if it doesn't exist
  • Policy Initiative = A group of related policies bundled together (e.g., "Security Baseline")
  • Policies can be applied at Management Group, Subscription, or Resource Group level
Azure Policy vs RBAC: Policy controls what resources can do (properties, locations, types). RBAC controls who can do things to resources (user permissions). They work together but are different systems.

Resource Locks

Prevent accidental deletion or modification of critical resources.

Lock TypeCan Read?Can Modify?Can Delete?
CanNotDeleteYesYesNo
ReadOnlyYesNoNo

Locks apply regardless of RBAC permissions — even Owners must remove the lock first before deleting.

Azure Blueprints

  • Define a repeatable package of: ARM templates + Policy assignments + RBAC assignments + Resource Groups
  • Unlike ARM templates, Blueprints maintain an active connection to what was deployed
  • Versioned, auditable, and trackable
  • Great for setting up new environments that must meet standards

Compliance Tools

  • Microsoft Service Trust Portal — Access audit reports, compliance guides, and trust documents about Microsoft cloud services
  • Microsoft Purview — Unified data governance: data catalog, data lineage, compliance management, risk management across your entire data estate
Policy = what resources can do. RBAC = who can do things. Locks = prevent accidental changes even by owners. Blueprints = repeatable compliant environments.

SLAs and Service Lifecycle

Azure Service Level Agreements (SLAs)

SLAs are formal documents describing Microsoft's uptime commitments for each Azure service.

SLAAllowed Downtime/YearAllowed Downtime/Month
99%~3.65 days~7.2 hours
99.9%~8.7 hours~43 minutes
99.95%~4.4 hours~22 minutes
99.99%~52 minutes~4.3 minutes
99.999%~5.3 minutes~26 seconds

Composite SLAs

When your app depends on multiple services, you multiply the individual SLAs:

App Service (99.95%) × SQL DB (99.99%) = 99.94%

The composite SLA is always lower than the individual SLAs. More dependencies = lower overall SLA.

How to Improve Your SLA

  • Availability Zones — Deploy across multiple zones in a region
  • Multiple regions — Deploy to more than one region
  • Redundancy / Fallback — Add a queue or cache as fallback if a service is unavailable
  • Choose higher-SLA services — Some tiers have better SLAs

If Azure Fails the SLA

You may be eligible for service credits — a percentage of your monthly fees. You typically must file a claim.

Azure Service Lifecycle

  • Preview — Service is available for testing but NOT covered by SLAs or support. Two types:
    • Private Preview — Available to select customers by request
    • Public Preview — Available to all Azure customers
  • General Availability (GA) — Service is fully released, covered by SLAs and support
Azure free account includes: $200 credit for 30 days + 12 months of popular free services + 25+ always-free services.

Frameworks & Best Practices

  • Cloud Adoption Framework (CAF) — Microsoft's end-to-end guidance for cloud adoption: Strategy → Plan → Ready → Adopt (Migrate/Innovate) → Govern → Manage
  • Well-Architected Framework — Five pillars for high-quality workloads: Reliability, Security, Cost Optimization, Operational Excellence, Performance Efficiency
For composite SLAs: multiply. For improving SLAs: add redundancy. Preview services have NO SLA. The five Well-Architected Framework pillars match the five Advisor categories.
